# 서브에이전트 사용 판단과 작업 지시문

## 1. 사용 여부 판단

이 프로젝트는 서브에이전트를 반드시 사용해야 하는 규모는 아니다. 초기 설계와 기본 프로젝트 구조 생성 단계에서는 메인 에이전트가 단독으로 진행하는 편이 일관성 측면에서 유리하다.

다만 구현 단계에 들어가면 기능 영역이 비교적 명확히 분리된다. 백엔드, 수강생 UI, UI/UX 검토, 인프라, QA/보안은 서로 다른 관점이 필요하므로 제한된 범위에서 서브에이전트를 사용하는 것이 좋다.

권장 판단:

| 단계 | 서브에이전트 사용 |
| --- | --- |
| 초기 설계 | 사용하지 않음 |
| 기본 Django 구조 생성 | 사용하지 않음 |
| 모델과 권한 구현 | 선택적 사용 |
| 수강생 화면 구현 | 선택적 사용 |
| UI/UX 품질 검토 | 사용 추천 |
| Docker/Synology 배포 | 사용 추천 |
| QA/보안 점검 | 사용 추천 |

## 2. 운영 원칙

- 메인 에이전트가 전체 아키텍처와 코드 일관성을 관리한다.
- 서브에이전트는 제한된 영역만 담당한다.
- 여러 서브에이전트가 같은 파일을 동시에 수정하지 않는다.
- 작업 전 담당 범위와 제외 범위를 명확히 한다.
- 작업 후 수정 파일, 구현 내용, 테스트 방법, 남은 문제를 보고한다.
- 메인 에이전트가 최종 통합, 리팩터링, 문서 정리를 담당한다.

## 3. backend-agent

### 역할

Django 기반 승인형 LMS의 백엔드 기능을 담당한다.

### 담당 범위

- 사용자 역할: 관리자, 수강생
- Course 모델
- Lesson 모델
- VideoAsset 모델
- Enrollment 모델
- WatchProgress 모델
- ReEnrollmentRequest 모델
- Django Admin 설정
- 수강 권한 확인 로직
- 수강 기간 확인 로직

### 제외 범위

- Bootstrap template 세부 구현
- Docker Compose 배포 구성
- 최종 통합 리팩터링

### 작업 지시문

```text
Django 승인형 LMS의 백엔드 모델, 관리자 등록, 권한 검사 서비스를 구현한다.
담당 파일은 accounts, courses, lessons, enrollments, progress, core/services 중심으로 제한한다.
Course, Lesson, VideoAsset, Enrollment, WatchProgress, ReEnrollmentRequest 모델을 만들고 Django Admin에서 관리 가능하게 한다.
수강 승인 상태, 시작일/종료일, 영상 접근 가능 여부는 core/services/access.py로 분리한다.
영상 파일은 직접 URL 노출을 피하고, 권한 확인 후 접근하는 구조를 고려한다.
작업 후 수정 파일, 마이그레이션, 테스트 방법, 남은 문제를 보고한다.
```

## 4. student-ui-agent

### 역할

수강생이 사용하는 Django template 화면을 구현한다.

### 담당 범위

- 회원가입
- 로그인
- 강의 목록
- 수강 신청
- 내 강의실
- 강의 상세
- 영상 시청 페이지
- 시험 응시 페이지
- 수료 확인 페이지
- 재수강 신청 페이지

### 제외 범위

- DB 모델 구조 변경
- Docker 배포 설정
- 관리자 dashboard 전체 설계

### 작업 지시문

```text
Bootstrap 5 기반으로 실제 업무에 사용할 수 있는 수준의 수강생 화면을 구현한다.
담당 파일은 templates, accounts/views.py, courses/views.py, enrollments/views.py, lessons/views.py, progress/views.py 중심으로 제한한다.
로그인 후 수강생은 내 강의실로 이동한다.
내 강의실은 승인된 강의, 신청 대기 강의, 종료된 강의를 구분한다.
각 강의 카드에는 강의명, 수강 상태, 시작일, 종료일, 남은 기간, 진도율, 이어보기 버튼을 표시한다.
영상 시청 화면은 데스크톱 2열, 모바일 1열 반응형으로 구성한다.
동적 워터마크는 전체화면에서도 보이도록 player wrapper fullscreen 구조를 사용한다.
빈 데이터 화면, alert 메시지, 폼 오류 표시를 모든 주요 화면에 반영한다.
작업 후 화면 URL, 수정 파일, 테스트 방법, 남은 문제를 보고한다.
```

## 5. ui-ux-agent

### 역할

전체 UI/UX 품질을 관리하고 Bootstrap 5 기반 디자인 일관성을 점검한다.

### 담당 범위

- 공통 레이아웃
- 상단 네비게이션
- 공통 메시지
- 빈 데이터 화면
- 관리자 대시보드
- 수강생 내 강의실
- 강의 상세
- 영상 시청
- 카드, 테이블, 폼, 버튼, 배지 스타일 통일
- 모바일 반응형 검토

### 제외 범위

- 핵심 DB 모델 변경
- 권한 로직 변경
- 배포 구성 변경

### 작업 지시문

```text
Bootstrap 5 기반 업무용 LMS UI/UX 품질을 점검하고 개선안을 구현한다.
기본 브라우저 스타일로 방치된 화면이 없게 한다.
관리자 화면은 데스크톱 업무 효율을 우선하고, 수강생 화면은 모바일 반응형과 다음 행동 명확성을 우선한다.
버튼, 카드, 테이블, 폼, 배지 스타일을 통일한다.
빈 데이터, 오류, 성공 메시지, 처리 중 상태를 주요 화면에 반영한다.
영상 시청 화면은 영상이 가장 크게 보이도록 하고, 차시 목록은 데스크톱에서 옆, 모바일에서 아래에 배치한다.
작업 후 각 화면의 UX 의도, 수정 파일, 테스트 방법, 남은 문제를 보고한다.
```

## 6. infra-agent

### 역할

Synology NAS에서 Docker Compose로 운영 가능한 배포 구성을 담당한다.

### 담당 범위

- Dockerfile
- docker-compose.yml
- PostgreSQL 컨테이너
- Django app 컨테이너
- Nginx reverse proxy
- static/media/private media 볼륨 구조
- `.env.example`
- DB 백업 스크립트 또는 절차
- 영상 파일 저장 경로 설계
- 운영 배포 문서

### 제외 범위

- Django 모델 구현
- 수강생 template 구현
- 시험 로직 구현

### 작업 지시문

```text
Synology NAS Docker Compose 운영 구성을 작성한다.
담당 파일은 Dockerfile, docker-compose.yml, nginx 설정, .env.example, deploy 문서로 제한한다.
PostgreSQL, Django/Gunicorn, Nginx 컨테이너를 구성하고 static, public media, private video volume을 분리한다.
NAS 절대경로는 환경변수 예시로만 제공한다.
DB 백업과 영상 파일 백업 절차를 문서화한다.
Synology Reverse Proxy HTTPS 방식과 컨테이너 Nginx HTTPS 방식을 비교해 추천한다.
운영 환경에서 DEBUG=False, ALLOWED_HOSTS, CSRF_TRUSTED_ORIGINS 설정을 고려한다.
작업 후 수정 파일, 실행 방법, 백업 방법, 남은 문제를 보고한다.
```

## 7. qa-security-agent

### 역할

권한, 영상 접근, 수강 기간, 워터마크, 관리자 기능을 QA/보안 관점에서 점검한다.

### 담당 범위

- 로그인하지 않은 사용자의 강의 접근 차단
- 수강 승인 전 영상 접근 차단
- 수강 시작일 이전 접근 차단
- 수강 기간 종료 후 영상 접근 차단
- 다른 수강생의 강의 접근 차단
- 영상 파일 직접 URL 접근 가능 여부
- 워터마크 표시 여부
- 전체화면 워터마크 표시 여부
- 모바일 화면 깨짐 여부
- 관리자 권한과 수강생 권한 분리
- 개인정보 노출 위험
- 관리자 페이지 접근 제한
- 수료증 다운로드 권한 검증

### 작업 지시문

```text
온라인 강의 사이트의 권한, 보안, 영상 접근, 워터마크, 관리자 기능을 점검한다.
코드 수정은 하지 않고 먼저 심각도 순서의 QA 리포트를 작성한다.
각 이슈에는 재현 방법, 영향도, 수정 제안을 포함한다.
확인 범위는 비로그인 접근, 승인 전 접근, 수강 기간 전후 접근, 타 사용자 데이터 접근, 영상 직접 URL, 전체화면 워터마크, 모바일 UI, 관리자 권한, 수료증 다운로드 권한이다.
필요한 수정이 명확하고 범위가 작을 때만 메인 에이전트 승인 후 수정한다.
```

## 8. 메인 에이전트 단독 진행 시 작업 순서

서브에이전트를 사용하지 않는 경우 다음 순서로 진행한다.

1. 기본 Django 프로젝트 구조 생성
2. custom User와 핵심 모델 구현
3. Django Admin 등록
4. 권한과 수강 기간 서비스 구현
5. 공통 Bootstrap template 구현
6. 수강생 화면 구현
7. 관리자 보조 화면 구현
8. 영상 스트림과 워터마크 구현
9. 시험과 수료증 구현
10. Docker Compose 배포 구성
11. QA와 보안 점검
12. 통합 리팩터링과 문서 정리
